# muduo

大二暑假实训期间写的。学完 socket 和 epoll 之后觉得光看书不太行，从网上找了个项目写写看看到底怎么回事。

结果搭出来了。从最底层的系统调用开始，一块一块往上摞，摞到最后在上面跑了个 HTTP 服务器。两个头文件，框架部分大概一千两百行，HTTP 部分大概五百行，没调任何现成的网络库。

## 搭的过程

### 系统调用封装

先写了个 Socket 类，把 socket fd 包起来，构造的时候创建，析构的时候关掉，RAII 那一套。Bind、Listen、Accept、Connect 这些常用操作都封进去了，还顺便加了设置非阻塞和地址重用的逻辑。一行 CreateServer 就能起个监听端口，不用每次写一堆模板代码。

然后是 Poller，对 epoll 的一层薄封装。epoll_create 创建实例，epoll_ctl 增删改事件，epoll_wait 阻塞等事件。里面维护了一个从 fd 到 Channel 指针的映射，等事件返回之后能根据 fd 找回对应的 Channel。

### 事件循环

Channel 代表一个文件描述符和它关心的事件。上面绑了五个回调——读到了干什么、能写了干什么、出错了干什么、连接断了干什么、任意事件触发干什么。调 EnableRead 就会把可读事件加到集合里，然后同步到 Poller。

EventLoop 是整个框架的心脏，就是一个死循环。每次循环做三件事：调 Poller 等事件、遍历就绪的 Channel 挨个调回调、处理别人通过队列丢过来的任务。构造函数里还偷偷创建了一个 eventfd 并注册到自己的 epoll 里——别的线程想叫醒这个 loop 的时候，往这个 eventfd 写一个字节，epoll 就返回了。

LoopThread 把 EventLoop 包进一个独立线程里。构造函数一调就开线程，线程里创建 EventLoop 然后进死循环。外面拿 LoopThread 的 GetLoop 拿到的就是已经跑起来的 loop。LoopThreadPool 就是一堆 LoopThread，新连接来了轮询分配。

### TCP 连接

Buffer 是自己写的读写缓冲区。底层是 vector，用两个下标标记读到哪写到哪。扩容的时候不是直接 resize——先看尾部剩多少空间，够就直接写；不够的话看头部（已经被读过的空间）加上尾部够不够，够的话把未读数据挪到开头，重置两个下标，把空闲空间并到尾部；还不够才扩容。这样能少很多次内存重新分配。还做了一个按行读的功能，底层用 memchr 找换行符，比一个个字符比快。

Connection 就是一个 TCP 连接。里面塞了五个东西：一个 Socket 管 fd、一个 Channel 管事件、两个 Buffer 分别收入站和出站数据、一个 Any 存上下文。

Acceptor 是监听器。一收到新连接就 accept，创建一个 Connection，从线程池取个 loop 丢过去。

TcpServer 把上面这些东西全串起来。对外就暴露几个回调：连接建立、收到消息、连接断开。空闲超时用时间轮管的——每次收到消息就刷新一下定时器，超时没动静自动断开。

时间轮是个环形数组，分了六十个格子，对应六十秒。timerfd 每秒触发一次，指针走一格，走到哪格就把那格清空。格子里的任务是用 shared_ptr 存的，清空的时候引用计数归零，析构函数里执行回调。加任务的时候按当前指针位置加上延迟秒数取模六十算出该放哪个格子，删除也就是把对应的 shared_ptr 置空，都是 O(1)。

Any 是一个类型擦除容器。里面一个抽象基类 holder，一个模板子类 placeholder，外加 clone 虚函数支持深拷贝。把任意类型塞进去，需要用的时候取出来，类型不对返回空指针而不是崩。Connection 用它来存上下文——HTTP 连接就往里塞 HTTP 解析器，Echo 连接就什么都不塞。

### HTTP 协议

HTTP 请求的解析是分阶段状态机。第一阶段读请求行，用正则把请求方法、路径、query string、HTTP 版本提取出来。第二阶段一行一行读头部字段，读到空行说明头部结束了。第三阶段根据 Content-Length 读请求体。第四阶段标记解析完成。每一阶段如果数据不够就直接 return，等下次数据到了从当前阶段继续，状态不会丢。整个状态机就挂在 Connection 的 Any 上下文里。

HTTP 服务器这边维护了四张路由表，分 GET、POST、PUT、DELETE，每张表里存的是正则表达式和处理函数的配对。请求来了先拿路径去对应方法的路由表里正则匹配，匹配上了就调处理函数，没匹配上就尝试当静态文件处理，再没有就返回 404。

静态文件那边做了几个安全处理：只响应 GET 和 HEAD 请求；检查路径防止目录穿越（把路径按斜杠分割，遇到 .. 就计深度，深度变为负数说明想跳出根目录，直接拒绝）；根据文件后缀查 MIME 类型填 Content-Type 响应头。

HTTP 响应是纯手写拼的。按 HTTP 协议规定的格式，先拼状态行，再拼头部，然后一个空行，最后接正文。根据请求头里的 Connection 决定响应完是保持连接还是关闭。

## 踩过的坑

调试 HTTP 服务器的时候浏览器一直转圈不显示内容，肉眼看了好几遍响应拼装的代码都觉得没问题。后来写了个小程序直接把发出的原始字节打印出来，发现头部和正文之间少了个回车换行——HTTP 协议要求 header 和 body 之间必须是一个空行也就是连续两个回车换行，我只写了一个。多这一个字符的事，肉眼根本看不出来，debug 程序两分钟就定位了。

还有一个坑是跨线程任务唤醒。一开始没加 eventfd，别的线程往 loop 里投任务只能等下次网络事件触发才处理，有时候等好几秒。加上 eventfd 之后立刻见效。

时间轮那里也踩过坑：某个槽位的回调里如果又往同一个槽位塞了新任务，会被紧接着的 clear 一起清掉。修法是先取出来再清空。

Buffer 扩容一开始直接 resize，浪费了头部已读过的空闲空间。后来改成先紧凑搬移——用 std::copy 把未读数据整体挪到 buffer 开头，重置读写下标——不够再扩容。

## 怎么跑

只需要 g++ 就行，没有第三方依赖。进 source/echo 目录 make 然后跑 main，另一个终端 nc localhost 8500 就能测试 Echo 服务器。进 source/http 目录 make 然后跑 main，浏览器访问 8085 端口就能看到 HTTP 服务器。

## 用到的

C++、epoll、fcntl、eventfd、timerfd、socket、pthread、正则表达式、HTTP/1.1
